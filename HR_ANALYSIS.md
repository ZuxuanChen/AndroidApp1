# LifeTrack — 项目亮点分析（资深 HR / 技术面试官视角）

## 项目概述
LifeTrack 是一款**个人全生命周期管理平台**，涵盖课程排期、任务追踪、目标管理、睡眠记录、习惯打卡、心情日记、专注计时、成就徽章等 8 大模块。采用 **Web + Android 双端架构**，纯本地存储、零后端依赖。

---

## 能让面试官眼前一亮的 8 个核心亮点

### 1. 全栈能力覆盖 — 不是玩具项目
- **Web 端**: React 19 + TypeScript + Vite + Tailwind CSS + Dexie.js (IndexedDB)
- **Android 端**: Kotlin + Jetpack Compose + Material Design 3 + Room + Hilt + MVI
- 一个人独立完成双端，证明技术栈广度与快速学习能力

### 2. 产品思维落地 — 不是堆砌功能
- 识别真实用户痛点：**课程时间冲突** → 实现重叠检测 + 确认覆盖
- **任务截止日期** → 过期告警 + 筛选排序 + 仪表盘横幅提醒
- **习惯打卡 streak** → 连续天数统计，利用心理学"损失厌恶"增强用户粘性
- 每个功能都有清晰的业务场景支撑

### 3. 数据安全与完整性 — 离线优先的严肃态度
- 纯 IndexedDB 本地存储，无服务器依赖，用户数据完全自主
- 实现完整的 **JSON 导出/导入** 备份系统：版本校验、数量预览、二次确认、ID 脱敏
- 修复 `completedAt` → `completedDates` 字段迁移 bug，体现对数据一致性的重视
- 全局 **ErrorBoundary** 崩溃保护 + 数据清除恢复机制

### 4. 测试纪律 — 102 个 edge case 测试
- 使用 vitest + @testing-library 搭建完整测试框架
- 覆盖：数据校验、时间计算、重叠检测、导入安全、 streak 逻辑、除零保护
- 测试文件 8 个，全部通过，TypeScript 零报错
- 说明候选人理解"代码可维护性"与"回归防护"的重要性

### 5. 工程化细节 — 代码质量可见
- **DRY 原则**: 提取共享 `COLORS` 常量到 `db.ts`，消除 4 个组件重复定义
- **键盘快捷键**: `Ctrl+1~8` 切换标签页，考虑效率型用户
- **实时状态**: Dashboard 课程卡片显示"进行中/已完成/已结束/即将开始"动态状态
- **输入防抖与校验**: 空标题拦截、60 字上限、时间逻辑校验、12 小时阈值警告

### 6. 迭代管理能力 — 版本化交付
- 建立 `v0.2.0-product-improvements` 分支，禁止直接修改 main
- 每个版本附带详细 CHANGELOG（功能/修复/测试/类型安全）
- 6 个版本递进：备份系统 → 截止日期 → 稳定性 → 进度可视化 → 除零修复 → 实时状态
- 体现项目管理的条理性和交付能力

### 7. 架构设计深度 — 不是表面功夫
- **分层架构清晰**: db.ts（数据层）→ 组件（表现层），状态管理通过 Dexie 表直接驱动
- **Android MVI 模式**: State/Intent/Effect 分离，UseCase 层解耦业务逻辑
- **类型安全**: TypeScript 严格模式，`tsc --noEmit` 零输出零错误
- **自定义类型**: `LessonId`, `StudentId`, `TutorResult<T>` 等 DDD 风格领域类型

### 8. 用户增长与留存设计 — 超出技术本身的思考
- **成就徽章系统**: 8 种徽章（早起鸟、肝帝、睡神、周冠军等），利用游戏化增强留存
- **连续打卡 streak**: 🔥 N 天连续，社交货币属性
- **数据可视化**: 7 天趋势图、工作量统计、饼图、进度条
- **主题色切换**: 用户可自定义 Dashboard 主题，增加情感连接

## 面试时建议强调的故事线

> "这个项目的起点是我自己的需求——作为学生需要同时管理课程、作业、习惯打卡和睡眠。我发现市面上的工具要么太重（Notion）、要么太单一（番茄钟），于是决定自己做一个**刚好够用**的平台。
>
> 过程中我遇到过一个真实的 bug：最初用 `completedAt` 字段标记课程完成，但发现同一天多次打开番茄钟会重复记录。我把它改成了 `completedDates` 数组，加了去重逻辑，还写了测试防止回归。
>
> 另一个让我自豪的决定是**坚持纯本地存储**。很多同类产品强制登录和云端同步，但我相信个人数据应该完全属于用户自己。为此我实现了完整的 JSON 备份/恢复系统，用户随时可以导出全部数据。
>
> 最后，我刻意控制了项目范围——不碰后端、不做登录、不上传。把有限精力集中在**用户体验的打磨**上，比如键盘快捷键、实时状态标签、过期任务告警。我认为好的软件不是功能多，而是每个功能都恰到好处。"
